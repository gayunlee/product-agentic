### web-agent

## us-plus Web MCP 인터페이스 탐색 결과

### 핵심 요약

us-plus의 에이전트 통신은 **SSE/WebSocket/REST가 아니라, 브라우저 내 Web MCP(Model Context Protocol)** 방식입니다. `navigator.modelContext`에 도구(tool)를 등록하면, 외부 에이전트가 Playwright로 브라우저에 접속해 이 도구를 호출하는 구조입니다.

---

### 아키텍처

```
┌─────────────────┐     check protocol     ┌──────────────────┐
│ Settings Agent   │ ───────────────────▶  │ QA/Verify Agent   │
│ (us-product)     │                        │ (us-explorer)     │
│ Admin API 읽기   │                        │ Playwright 실행   │
└─────────────────┘                        └────────┬─────────┘
                                                     │ navigator.modelContext
                                                     ▼
                                            ┌──────────────────┐
                                            │   us-plus (Web)   │
                                            │  Web MCP Tools    │
                                            │  (순수 클라이언트) │
                                            └──────────────────┘
```

**서버 통신 없음** — 에이전트가 브라우저를 직접 열고, DOM에 등록된 MCP 도구를 호출합니다.

## us-explorer-agent: Web MCP 통신 방식 정리

### 아키텍처 개요

```
세팅 에이전트 → POST /verify 또는 /verify/agent
    → Python check 함수들
    → Playwright 브라우저 (싱글톤)
    → navigator.modelContext.callTool()  ← Web MCP 인터페이스
    → us-plus 프론트엔드의 MCP 도구 실행
    → JSON 응답 파싱 & 반환
```

**핵심 포인트:** SSE/WebSocket이 아니라 **Playwright를 통한 브라우저 내 JavaScript 평가** 방식으로 Web MCP와 통신합니다.

---

### 1. API 엔드포인트 (server.py)

| 엔드포인트 | 메서드 | 설명 |
|---|---|---|
| `/health` | GET | 헬스체크 `{"status": "ok", "service": "us-explorer-agent"}` |
| `/verify` | POST | 결정론적 검증 (LLM 없이 expected vs actual 비교) |
| `/verify/agent` | POST | LLM 에이전트 기반 검증 (Strands Agent + Bedrock) |

---

### 2. 요청/응답 포맷 (models.py)

**요청 (`VerifyRequest`):**
```json
{
  "context": {
    "master": {"cmsId": "35", ...},
    "products": {...}
  },
  "checks": [
    {
      "tool": "check_subscribe_page",
      "args": {"product_group_code": 148},
      "expected": {"found": true, "isButtonActive": true}
    }
  ]
}
```

**응답 (`VerifyResponse`):**
```json
{
  "results": [
    {
      "tool": "check_subscribe_page",
      "status": "pass",          // "pass" | "fail" | "error"
      "expected": {"found": true},
      "actual": {"found": true, "isButtonActive": true, ...},
      "mismatches": [],
      "error": null
    }
  ],
  "summary": {"total": 2, "passed": 1, "failed": 0, "errors": 0},
  "context": {...}
}
```

### 3. Web MCP 통신 방식 (webmcp_client.py)

**SSE가 아닌 Playwright `page.evaluate()` 기반:**

```python
async def call_tool(name: str, arguments: dict | None = None) -> dict:
    # 브라우저 내에서 JS 실행
    result = await page.evaluate("""
        navigator.modelContext.callTool({
            name: "get_displayed_clubs",
            arguments: {"productGroupCode": 148}
        })
    """)
    # result.content[0].text → JSON 파싱 후 반환
```

**MCP 응답 구조:**
```json
{
  "content": [{"text": "{\"clubs\": [{...}]}"}]
}
```
`content[0].text`가 JSON **문자열**(객체가 아님)이라 한 번 더 파싱이 필요합니다.
