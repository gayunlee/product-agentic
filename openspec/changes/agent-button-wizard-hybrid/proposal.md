# 버튼 위저드 + Agent 하이브리드 아키텍처

## 문제

현재 모든 요청이 오케스트레이터 LLM → executor LLM 경로를 타면서:
- 단순 실행 (비공개/공개/히든)도 LLM 8회 호출 = 28초
- Bedrock API 왕복 2-3s × LLM 라운드 수가 병목
- 유저가 "1번", "네" 같은 단순 선택을 해도 LLM이 해석

## 핵심 인사이트

시나리오 33개 + Langfuse 574 트레이스 분석 결과:

1. **executor가 하는 일의 대부분은 "순서대로 API 호출 + 유저 선택 받기"** — LLM의 적응적 판단이 필요한 건 첫 턴(의도 파악)뿐
2. **후속 턴(선택, 확인)은 구조화된 입력** — 버튼 클릭으로 대체 가능
3. **response.py에 버튼 인프라가 이미 존재** — select, confirm, complete 템플릿 + 버튼 패턴

## 제안

**빈번한 단순 실행 → 버튼 위저드 (LLM 0~1회)**
**복잡한/모호한 요청 → Agent (LLM 필요한 만큼)**

```
현재:  턴1 LLM → 턴2 LLM → 턴3 LLM → 턴4 LLM   = 28초
버튼:  턴1 LLM → 턴2 코드 → 턴3 코드 → 턴4 코드  = ~5초 + 클릭 대기
```

## 범위

### In Scope
- 버튼 위저드 패턴 정의 (빈번한 실행 시나리오)
- 오케스트레이터가 버튼 입력과 자연어 입력을 구분하는 라우팅
- 위저드 내 이전/취소 버튼으로 플로우 제어
- executor LLM은 유지 (복잡한 요청의 fallback)

### Out of Scope
- 프론트엔드 (사이드패널) UI 구현
- 프롬프트 캐싱 / 스트리밍 (별도 Phase)
- 새로운 API 추가

## 예상 효과

| 시나리오 | 현재 | 버튼 위저드 |
|---------|------|----------|
| 상품 비공개 | 28초 (LLM 8회) | ~5초 (LLM 1회) |
| 상품페이지 비공개 | ~20초 (LLM 6회) | ~4초 (LLM 1회) |
| 노출 진단 | 6.75초 (이미 코드화) | 유지 |
| "왜 안 보여?" (자연어) | ~7초 | 유지 (Agent 처리) |
| "비공개하면 다른거 공개돼?" | ~15초 | 유지 (Agent 처리) |
