## 1. CoT 진행 상황 노출

### 현재 상태
- SSE `event: progress`로 tool 호출 진행 상황 전송 중
- 프론트에 `sidePanelLoadingTextAtom` 있음
- 하지만 실제로는 고정 메시지 로테이션 ("생각하는 중..." → "정보 조회 중...")

### 변경
- SSE progress 이벤트를 패널이 실시간으로 표시
- 에이전트가 실제로 호출하는 tool에 맞는 메시지

```
"🔍 오피셜클럽 검색 중..."  ← search_masters 호출
"📦 상품 조회 중..."         ← get_product_list_by_page 호출
"✅ 사전조건 확인 중..."     ← check_prerequisites 호출
```

### 구현
- 서버: 이미 `TOOL_PROGRESS` 매핑 있음. 변경 없음
- 패널: SSE `progress` 이벤트 수신 → 로딩 텍스트 업데이트

**난이도: 낮음 (패널만)**

## 2. 메시지 큐잉

### 개념
유저가 에이전트 응답 대기 중에 추가 메시지를 미리 작성 → 큐에 쌓고 순차 처리.

### 동작

```
유저: "조조형우 상품 비공개해줘"     → 전송, 응답 대기 중
유저: "그리고 봄날의 햇살도 확인해줘" → 큐에 추가 (UI에 "대기 중" 표시)

에이전트 응답 1 도착 →
  → 큐에서 다음 메시지 자동 전송
  → "봄날의 햇살도 확인해줘" 처리 시작

에이전트 응답 2 도착 → 완료
```

### 구현
- 패널 내부에 messageQueue 상태
- 입력 비활성화 대신, 입력은 열어두되 큐에 쌓기
- 큐에 메시지 있으면 "1개 대기 중" 표시
- 응답 도착 시 큐에서 다음 자동 발송

**난이도: 낮음 (패널만)**

### 주의사항
- 위저드 진행 중에는 큐잉 비활성화 (버튼만 받으니까)
- 큐 취소 기능 필요 (대기 중인 메시지 삭제)

## 3. 위저드 확장 (Phase 3 결과 활용)

Phase 3에서 선언적 위저드가 되면, 빈출 시나리오를 빠르게 등록:

| 위저드 | YAML | 예상 효과 |
|--------|------|----------|
| 상품페이지 비공개 | product_page_inactive.yaml | 28초 → ~5초 |
| 상품페이지 공개 | product_page_active.yaml | ~20초 → ~4초 |
| 상품 비공개 | product_hide.yaml | ~20초 → ~4초 |
| 상품 공개 | product_show.yaml | ~15초 → ~4초 |
| 히든 처리 | hidden.yaml | ~20초 → ~4초 |
| 노출 진단 | diagnose.yaml | 이미 코드화, 위저드로 통합 |

**런칭 체크 외 6개 위저드를 YAML만 작성해서 추가.**

## 4. (후속) 투기적 실행

Phase 4 이후, 별도 설계 필요.

### 개념
모호한 요청 → 가능한 분기를 병렬로 미리 조회 → 유저 확인 후 맞는 결과만 사용.

### 검토 필요
- executor가 "모호함 감지 → 병렬 조회" 판단을 어떻게 하는가
- 불필요한 API 호출 비용 vs 속도 개선 트레이드오프
- 병렬 tool 실행 지원 (Strands SDK 제약)
- 결과 캐싱/폐기 전략

### 이 Phase에서는 설계하지 않음.
